# Remudero — Production Readiness Review (fresh-eyes recon)

**Date:** 2026-07-21 · **Reviewer:** external recon pass, no project history · **Method:** 7 parallel source audits + direct measurement of the live clone, state dir, launchd unit, and GitHub branch protection. Every finding is labeled **OBSERVED** (ran it / read it at file:line) or **INFERRED** (reasoned; names its verifier). Backlog (`tasks.yaml`, `tasks.d/`, `MASTER-PLAN.md`, `LEARNINGS.md`, `DECISIONS.md`) was read **only after** findings were frozen, per the sequencing rule, to measure the backlog's blind spots.

**Repo scale (OBSERVED):** 328 commits (323 merged PRs, PRs up to #466), 65k LOC (23.5k src, 31k test), 1703/1703 tests pass in 22s, depcruise 0 violations, jscpd 1.80%, `npm ci` 0 vulnerabilities, tsconfig `strict:true` with 0 real `any`.

## 1. Executive summary

Remudero is an **unusually disciplined** self-hosting harness. The engineering culture is real and measurable: a deterministic model-per-rung policy behind a single SDK choke point, a fail-closed plan linter that runs before any spend, ratchet gates for coverage/mutation/duplication/complexity, SHA-pinned CI actions with least-privilege permissions, and — most striking — a whole lineage of fixes that convert "cannot observe → treat as absent/permitted" into "mark indeterminate, defer." This review's own ordering discipline confirms the culture: **25 of 36 findings were already at least partially tracked** in the backlog before I read it.

It is nevertheless **not yet safe to run fully unattended**, for three reasons that compound:

1. **The verification floor is defeatable by the very worker it governs.** The required merge status `remudero-review` is a plain GitHub commit status that the reviewed worker can POST itself (`gh` runs outside the sandbox with operator credentials; the deny-floor does not block `gh api …/statuses`), branch protection requires **no independent review** (confirmed live: no `required_pull_request_reviews`, `enforce_admins:false`), and a single PR can edit both the gate definition (`ci-gate.yml` REQUIRED array) and the ratchet baselines it is measured against. On top of that, the *default* merge path for any task not marked `tdd:strict` passes on a 34%-keyword prose match with **zero** proofs executed. The ceiling (tdd:strict + executed proofs) is strong; the floor is prose. **[R-2, R-3, R-6, R-13, R-14]**

2. **The daemon is in an active crash-loop right now.** An uncaught `execFileSync("gh issue create")` inside `escalate()` throws on GitHub rate-limit, kills the daemon, launchd `KeepAlive` relaunches it in ~60s, and the in-memory escalation-dedup set resets — so it re-escalates and re-exhausts quota. **OBSERVED: 460 `daemon.boot` ledger lines, 438 since Jul 19, one per minute at 04:02–04:13.** This is self-inflicted and burns the same GitHub quota the harness needs to function. **[R-1]**

3. **Operator-surface and worker security are weak.** `rmd serve` binds all interfaces (not localhost as it claims) behind a single static bearer token; **both tokens are printed in plaintext to a world-readable `serve.log` and the write token is embedded in the console URL as a query param**; a spawned worker holds the operator's real GitHub token with a blast radius covering `gh pr merge`, workflow edits, non-force push to main, and `gh gist` exfiltration. **[R-4, R-5, R-3]**

**The three findings that matter most:** R-6 (self-modifiable gates with no independent reviewer — confirmed live), R-1 (live daemon crash-loop), and the ledger triad R-8/R-9/R-16 (the ledger is simultaneously the audit log and the backing store for the dispatch circuit-breaker and sweep dedup, yet it is unlocked against concurrent appends, unbounded, and its absence silently zeroes the breaker).

**The single biggest risk:** **R-6.** For a system whose explicit purpose is to merge its own code, a verification gate that the system can weaken in a PR that nothing independent reviews is the one structural hole that makes every other gate contingent. Everything else is a bug; this is a missing invariant.

## 4. Top 10 recommendations (ranked by severity × leverage)

Each carries **emitter-ready acceptance criteria** in falsifier shape ("X under Y FAILS") with a named unit test, so the operator can file them verbatim.

### T1 — Make the verification gate un-weakenable by the system it governs *(R-6, R-3, R-2; addresses the single biggest risk)*
The merge floor must have an independent anchor the reviewed worker cannot forge or edit in-band.
- **AC1:** A PR that modifies `.github/workflows/ci-gate.yml`, `ci.yml`, or any `scripts/*-baseline.json` **without** a matching approval from an identity distinct from the PR author FAILS to satisfy branch protection. *proof:* `unit test: test/gate-self-modification.test.ts` asserts branch-protection config requires reviews for those paths (or a CODEOWNERS-gated required review).
- **AC2:** A `remudero-review` `success` status posted by a worker identity (not the orchestrator's dedicated reviewer identity/token) is **not** honored as the required check. *proof:* `unit test: test/review-status-provenance.test.ts` — a status whose poster ≠ the trusted reviewer app is treated as absent.
- **AC3:** A non-`tdd:strict` task with `executedCount===0` that arms auto-merge FAILS. *proof:* `unit test: test/review.test.ts` — `decideAutoMergeArm(capped, tddStrict=false)` returns `arm:false` (raise the floor, or require a global minimum executed-proof count).
- **AC4:** `deny-floor.sh` that permits `gh api …/statuses/… -X POST` or `gh pr merge` from a worker FAILS. *proof:* `unit test: test/deny-floor.test.ts`.

### T2 — Stop the daemon crash-loop *(R-1; live incident)*
- **AC1:** An `escalate()` whose `gh issue create` throws that propagates far enough to exit the daemon loop FAILS. *proof:* `unit test: test/escalate-nonfatal.test.ts` — inject a throwing issue gateway; assert the daemon loop continues and logs `escalation.failed`.
- **AC2:** The daemon booting more than 5 times in any rolling 10-minute window FAILS a health invariant. *proof:* `unit test: test/daemon-crashloop.test.ts` over a synthetic `daemon.boot` ledger; also a launchd `ThrottleInterval`.
- **AC3:** Escalation dedup that resets across a process restart FAILS. *proof:* `unit test: test/escalation-dedup-persist.test.ts` — dedup state is read from the ledger, not an in-memory Set.

### T3 — Close the token-leak and console-exposure holes *(R-5, R-4)*
- **AC1:** `serve.log` (or any persisted log) containing a 64-hex token FAILS. *proof:* `unit test: test/serve-no-token-in-log.test.ts`.
- **AC2:** The console URL handed to the operator embedding the **write** token FAILS (hand out read scope; require header for writes). *proof:* `unit test: test/serve-console-url.test.ts`.
- **AC3:** `server.listen` bound to any address other than `127.0.0.1` / the configured tailnet IP FAILS. *proof:* `unit test: test/serve-bind.test.ts`.

### T4 — Make ledger writes atomic and decision-safe *(R-8, R-16, R-17)*
- **AC1:** Two concurrent `appendLedger` calls producing a torn or dropped line FAILS. *proof:* `unit test: test/ledger-atomic.test.ts` — N concurrent appenders, assert line count and JSON-validity of every line.
- **AC2:** The dispatch circuit-breaker count changing when the ledger is rotated/relocated FAILS. *proof:* `unit test: test/breaker-survives-rotation.test.ts`.
- **AC3:** A reader observing a half-written `status.json` regressing a merged task to `queued` FAILS. *proof:* `unit test: test/status-atomic-write.test.ts` — temp-file + `rename`.

### T5 — Ledger compaction/archival, breaker-safe *(R-9, coupled to T4/R-16)*
- **AC1:** `ledger.ndjson` exceeding a size ceiling with no archived roll present FAILS. *proof:* `unit test: test/ledger-rotation.test.ts`.
- **AC2:** A rotation that drops a decision-relevant line (dispatch/escalation/credit) from the breaker/dedup reader's view FAILS. *proof:* same test — a compacted "decision tail" must remain visible to `readLedgerLines`.
- **AC3:** `sweep.credit_backfill` emitting a ledger line for an `already_credited` candidate FAILS. *proof:* `unit test: test/credit-backfill-quiet.test.ts` (kills ~5k no-op lines).

### T6 — Close the headroom fail-open with a bounded fallback *(R-7)*
- **AC1:** `readUsage()` returning null AND the daemon spawning a task in the same tick FAILS, unless a bounded degraded-mode budget applies. *proof:* `unit test: test/daemon-headroom-failopen.test.ts`.
- **AC2:** N consecutive `headroom.unavailable` readings without entering a slowed/paused mode FAILS. *proof:* same test with a synthetic unreadable-usage sequence.

### T7 — Fence untrusted text before it reaches write-capable worker prompts *(R-19, R-20)*
- **AC1:** A CI-log tail containing an injected instruction reaching a fix-worker prompt un-fenced FAILS. *proof:* `unit test: test/fix-prompt-fencing.test.ts` — untrusted content is wrapped in an inert, labeled block.
- **AC2:** A GitHub issue body reaching the inbox draft prompt un-fenced FAILS. *proof:* `unit test: test/inbox-draft-fencing.test.ts`.

### T8 — Promote security scanners to blocking + fix the OSV race *(R-11)*
- **AC1:** A PR with an OSV-detected CVE that merges FAILS. *proof:* `unit test: test/osv-blocking.test.ts` (planted-CVE fixture) — `osv-scanner-pr` added to `ci-gate` REQUIRED so it is awaited, not raced.
- **AC2:** `semgrep`/`dependency-review` marked `continue-on-error` while claimed as gates FAILS a config assertion. *proof:* `unit test: test/scanner-gate-config.test.ts`.

### T9 — Per-diff coverage + widen the ratchet-config awareness surface *(R-12, R-15, R-10)*
- **AC1:** New source lines added with zero covering tests that merge above the aggregate floor FAILS. *proof:* `unit test: test/diff-coverage.test.ts`.
- **AC2:** A diff lowering a coverage/mutation baseline or editing a CI workflow that passes the review floor without a flagged rationale FAILS. *proof:* `unit test: test/awareness-surface-ratchet.test.ts` (add those paths to `USER_VISIBLE_SURFACE_RE`).

### T10 — Operator runbook + log rotation + backup *(R-30, R-25, R-31; the "system is un-operable in a crisis" cluster)*
- **AC1:** A docs-freshness check where README/CONTRIBUTING/operator-guide contradict code (required-check name, `--repo`, WS-stage claim) FAILS. *proof:* `unit test: test/docs-claims.test.ts` (extend the existing claims-check to hand-written docs).
- **AC2:** Any of daemon.err/out, serve.log, drain.log lacking a size/age rotation policy FAILS. *proof:* `unit test: test/log-rotation.test.ts` or a `newsyslog.d` fixture.
- **AC3:** A missing runbook entry for {daemon restart via launchctl, token rotation, `ANTHROPIC_*` boundary, first-run setup, quiet-hours} FAILS a runbook-coverage check. *proof:* `unit test: test/runbook-coverage.test.ts`.

## 6. Process proposals (Phase 5) + the audit-rung design

### Missing organs
1. **Ledger compaction/archival (R-9)** — none exists. Must be **breaker-safe** (R-16): roll to `ledger-YYYY-MM.ndjson.gz`, keep a compacted decision-relevant tail (dispatch/escalation/credit lines) that `readLedgerLines` still sees, else rotation zeroes the circuit breaker.
2. **Log rotation (R-25)** — none. `newsyslog.d` entry or in-code size cap for all four logs; note serve.log *also* leaks tokens (R-5) so rotation ≠ sufficient.
3. **Backup & restore of `state/`** — MASTER-PLAN §10 WS-7 *describes* "nightly `$ROOT/state` backup with a documented + drilled restore," but **grep finds zero filed tasks** — it is described, unbuilt, unfiled. The ledger is the irreplaceable artifact (credit history + circuit-breaker state); `status.json` is rederivable. If `state/` is lost, a merged task can re-dispatch (R-16). If the Mac dies, the repo survives on GitHub but operator retro/learnings state does not. **File it.**
4. **Disaster-recovery drill** — well-tracked (W1-T147 chaos drill, W1-T12c/d/e crash-recovery). Extend to a `state/`-loss scenario.
5. **Dependency-update cadence** — tracked (dependabot.yml SHA-pinned weekly; W1-T54/T55). Gap: acting on advisory OSV/dep-review findings (R-11).
6. **Docs-freshness for hand-written narrative** — generated docs are gated; README/CONTRIBUTING/operator-guide are not (R-30). Extend `claims-check` to them.
7. **Recurring self-audit** — designed below.

### The audit rung — `rmd audit`

**Purpose:** re-run a scoped version of *this* review on a cadence; route findings into the ratification inbox as `P##` proposals; never re-report accepted risks.

**Scope tiers (cheap → deep):**
- **T0 SMOKE** — every daemon idle tick, deterministic, **no model**. Assert invariants: ledger size < ceiling; worktrees dir < ceiling; branch count < ceiling; no dead-pid inflight lock older than N h; `serve.log` contains no token; **daemon boot-rate < K/hour (catches R-1)**; `headroom.unavailable` ratio < threshold over 24 h. Emits `audit.smoke`, escalates on breach. ~free.
- **T1 WEEKLY** — one sonnet recon pass, scoped: docs-vs-reality diff on hand-written docs; new-since-last-audit fail-open catch-block scan (grep the diff); `run-task.ts` LOC-growth delta; CVE/dependency delta.
- **T2 MONTHLY** — the full 8-pillar fan-out (this review's shape, ~7 workers + synthesis), producing `recon-YYYY-MM-DD.md`.
- **T3 ON-DEMAND** — `rmd audit --deep`.

**Cadence wiring:** T0 piggybacks the daemon poll hook (`daemon.ts:274`, beside `deps.sweep`) — **the same missing hook R-24's retro needs; build one scheduler for both.** T1/T2 need a launchd calendar trigger.

**Findings flow:** audit writes to `inbox-proposals.json` as `P##` proposals (not direct tasks), so operator `rmd approve` gates landing — reuses the P25 ratification machinery already built.

**Avoiding re-report of accepted risk:** maintain `state/audit-accepted.json` — a set of finding fingerprints (`pillar + file + rule-slug`) the operator marked accepted/won't-fix/by-design, each with a rationale and an **expiry**. Audit suppresses a fingerprint until expiry; a finding whose evidence *changes* (severity up, new file) re-surfaces flagged "changed since accepted." This is the ratchet-baseline pattern applied to findings.

**Fixture #1:** *this file* (`recon-2026-07-21.md`) is the golden input. The T2 audit's first acceptance test: "reproduces ≥ 80% of these 36 findings from source."
